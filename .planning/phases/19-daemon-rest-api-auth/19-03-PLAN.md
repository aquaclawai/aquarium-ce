Plaintext is shown exactly once at creation; all subsequent reads expose only the hashed projection. Revoke is soft (sets `revoked_at`) and takes effect on the next daemon request (≤1s, validated in 19-04 E2E).

    ```typescript
    import { Router } from 'express';
    import { requireAuth } from '../middleware/auth.js';
    import {
      issueDaemonToken,
      listDaemonTokens,
      revokeDaemonToken,
    } from '../services/daemon-token-store.js';
    import type { ApiResponse, DaemonToken, DaemonTokenCreatedResponse } from '@aquarium/shared';

    const router = Router();
    router.use(requireAuth);

    // CE: single default workspace (matches routes/runtimes.ts, routes/agents.ts, routes/issues.ts).
    // TODO(EE): swap for `req.auth.workspaceId` once the auth payload carries it.
    const DEFAULT_WORKSPACE_ID = 'AQ';

    // ── POST / — create token (plaintext returned ONCE) ────────────────────
    router.post('/', async (req, res) => {
      try {
        const name = typeof req.body?.name === 'string' ? req.body.name.trim() : '';
        if (!name) {
          res.status(400).json({ ok: false, error: 'name required' } satisfies ApiResponse);
          return;
        }
        if (name.length > 100) {
          res.status(400).json({ ok: false, error: 'name too long (max 100 chars)' } satisfies ApiResponse);
          return;
        }
        const expiresAt = typeof req.body?.expiresAt === 'string' ? req.body.expiresAt : null;
        const userId = req.auth?.userId ?? null;

        const payload = await issueDaemonToken({
          workspaceId: DEFAULT_WORKSPACE_ID,
          name,
          expiresAt,
          createdByUserId: userId,
        });

        res.json({ ok: true, data: payload } satisfies ApiResponse<DaemonTokenCreatedResponse>);
      } catch (err) {
        const msg = err instanceof Error ? err.message : String(err);
        res.status(500).json({ ok: false, error: msg } satisfies ApiResponse);
      }
    });

    // ── GET / — list (NEVER returns plaintext or token_hash) ───────────────
    router.get('/', async (_req, res) => {
      try {
        const tokens = await listDaemonTokens(DEFAULT_WORKSPACE_ID);
        // Service already returns DaemonToken projection (rowToDaemonToken from 19-01).
        // Defence-in-depth: the projection has no tokenHash/plaintext fields by construction.
        res.json({ ok: true, data: tokens } satisfies ApiResponse<DaemonToken[]>);
      } catch (err) {
        const msg = err instanceof Error ? err.message : String(err);
        res.status(500).json({ ok: false, error: msg } satisfies ApiResponse);
      }
    });

    // ── DELETE /:id — soft revoke ──────────────────────────────────────────
    router.delete('/:id', async (req, res) => {
      try {
        // Verify token exists in this workspace BEFORE calling revoke.
        // revokeDaemonToken returns false both for "not found" and "already revoked".
        // We want to distinguish: 404 for not-found, 200 for idempotent re-revoke.
        const tokens = await listDaemonTokens(DEFAULT_WORKSPACE_ID);
        const exists = tokens.some((t) => t.id === req.params.id);
        if (!exists) {
          res.status(404).json({ ok: false, error: 'token not found' } satisfies ApiResponse);
          return;
        }
        await revokeDaemonToken(req.params.id, DEFAULT_WORKSPACE_ID);
        // Swallow the boolean — idempotent by design.
        res.json({ ok: true, data: { ok: true } } satisfies ApiResponse<{ ok: boolean }>);
      } catch (err) {
        const msg = err instanceof Error ? err.message : String(err);
        res.status(500).json({ ok: false, error: msg } satisfies ApiResponse);
      }
    });

    export default router;
    ```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser cookie user → `/api/daemon-tokens/*` | Authenticated; user can only create/list/revoke tokens in their own workspace |
| Route → `daemon_tokens` DB | Parameterised knex via `daemon-token-store` service; no direct DB from route |
| HTTP response body → client | Plaintext MUST appear exactly once (POST response); all other responses expose only `DaemonToken` projection |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-19-13 (AUTH1) | Elevation of Privilege | adt_* bearer attempting user routes | mitigate | 19-01's `requireAuth` patch rejects `^Bearer\s+adt_` before cookie check. Verified here by test 5. |
| T-19-14 (AUTH2) | Information Disclosure | Plaintext leak on GET | mitigate | `listDaemonTokens` returns `DaemonToken` projection (no `tokenHash`, no plaintext); route returns it verbatim. Test 6 asserts no field contains `/adt_/` or `tokenHash`. |
| T-19-15 (AUTH3) | Elevation of Privilege (staleness) | Revoked token still valid | mitigate | `DELETE /:id` sets `revoked_at` via single UPDATE; `requireDaemonAuth` (19-01) excludes revoked rows via `.whereNull('revoked_at')` on every request. Test 12 asserts <1s SLA. |
| T-19-16 | Tenant Boundary Violation | Cross-workspace revoke | mitigate | `revokeDaemonToken(id, workspaceId)` filters by both id AND workspace_id in the UPDATE. Route pre-check via `listDaemonTokens` enforces same boundary for 404 distinction. Test 10 asserts behaviour. |
| T-19-17 (AUTH4) | Information Disclosure | Plaintext leakage after creation | mitigate | No re-regenerate endpoint; plaintext visible only in the POST response body (one-time). Subsequent GET never re-exposes it (service returns DaemonToken projection). 19-RESEARCH §"Why not also a regenerate endpoint" — rotation is revoke-then-create, two distinct operations. |
| T-19-18 | Input Validation | Name too long / missing | mitigate | 400 with explicit error for empty name or `> 100 chars`. `daemon_tokens.name` column is `string(100) notNullable` — DB backstop. Tests 3, 4. |
| T-19-19 (CE3) | Denial of Service | Token-management spam | accept | Bound by existing static `/api/` limiter (300 req / 15min) which DOES apply to `/api/daemon-tokens/*` (only `/api/daemon/*` is exempted per 19-02). Acceptable: human UI traffic, not high-volume. |
