function readVersion(): string {
  try {
    const here = dirname(fileURLToPath(import.meta.url));
    const pkgPath = join(here, '..', 'package.json');
    const pkg = JSON.parse(readFileSync(pkgPath, 'utf8')) as { version?: string };
    return pkg.version ?? '0.0.0';
  } catch {
    return '0.0.0';
  }
}

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Local git repo -> origin/main | Publishing code to the authoritative remote triggers CI + release workflows. Any tampering with the commit graph between the pre-push checks and the tag push could publish code that was NOT verified. |
| CI runner -> npm registry | release.yml's `npm publish` uses `NODE_AUTH_TOKEN: ${{ secrets.NPM_TOKEN }}`. The token has publish scope on @aquaclawai; its compromise could push arbitrary versions. |
| CI runner -> GHCR | docker/build-push-action pushes to `ghcr.io/aquaclawai/aquarium` using the default `GITHUB_TOKEN` with `packages: write`. Token is ephemeral per-run and cannot be exfiltrated without modifying the workflow itself. |
| Tagged commit -> published artifact | The tag is the sole input to release.yml. Once pushed, release.yml runs against `${{ github.ref_name }}` which is the tag itself — so the published artifact's source is traceable via `git show v1.4.0` forever. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-26-05-01 | Spoofing (tag points at wrong commit) | `git tag -a v1.4.0` | mitigate | Task 1 lands the version-bump commit on HEAD; Task 2 creates the annotated tag immediately afterward. The checkpoint's `how-to-verify` Step 2 explicitly greps `git log --oneline -1` for `chore: bump version to 1.4.0`, so a human confirms the tag-target commit before the push. |
| T-26-05-02 | Tampering (force-push erases pre-push checked code) | `git push origin main --tags` | mitigate | Checkpoint instructions explicitly say NEVER force-push `main`. If non-fast-forward, operator must rebase/merge FIRST and re-run pre-push checks. The checkpoint is human-gated so an autonomous executor cannot issue `--force`. |
| T-26-05-03 | Repudiation (release not traceable) | published artifact | mitigate | Annotated tag (`git tag -a v1.4.0 -m "..."`) carries the tagger identity + timestamp. release.yml's `uses: actions/checkout@v4` checks out the exact tag SHA. npm publish surfaces the ref in the published package-lock. The release log records the tag SHA + the gh run URL, so every step is auditable. |
| T-26-05-04 | Info disclosure (secrets in published tarball) | `npm publish` | mitigate | apps/server/package.json's `files: ["dist/"]` limits the tarball to built artifacts; `npm pack --dry-run` before the real publish lists the file set. Verification: run `npm pack --dry-run --workspace apps/server` locally pre-push and confirm no `.env`, `.planning/`, `tests/`, `src/` entries. This is added as a pre-push check in Task 1's release log. |
