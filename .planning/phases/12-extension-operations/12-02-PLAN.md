function buildSkillTogglePatch(
  skillId: string,
  enabled: boolean,
): Record<string, unknown> {
  return {
    skills: {
      entries: {

      },

**Refactor `enableSkill`:**

Replace the direct `skills.update` RPC call (line 274) with patchGatewayConfig + skills.status verification:

```typescript
// Gateway-first: config.patch to enable the skill (no SIGUSR1 -- skills are dynamically loaded)
const patch = buildSkillTogglePatch(skillId, true);
await patchGatewayConfig(instanceId, userId, patch, `Enable skill: ${skillId}`);

// Verify immediately via skills.status RPC (no reconnect wait needed -- no restart)
try {
  const statusResult = await gatewayCall(instanceId, 'skills.status', {}, 15_000) as {
    skills?: Array<{ name?: string; enabled?: boolean }>;
  } | null;
  const skillInGateway = statusResult?.skills?.find(s => s.name === skillId);
  if (skillInGateway && !skillInGateway.enabled) {
    // Gateway reports skill still disabled -- something went wrong
    console.warn(`[skill-store] Skill ${skillId} still disabled in gateway after enable config.patch`);
  }
} catch (verifyErr: unknown) {
  // Verification failure is non-fatal -- DB is already updated, config.patch succeeded
  console.warn(`[skill-store] skills.status verification failed after enable for ${skillId}:`, verifyErr);
}

// DB update (already handled before config.patch, keep as-is)
```

Wait -- looking at the current code more carefully, the DB update currently happens AFTER the RPC call. For the gateway-first model, keep the same order: config.patch first, then DB update. This is consistent with how patchGatewayConfig works (it persists to DB internally via deep-merge). However, patchGatewayConfig already persists the config to DB. The separate `db('instance_skills').update` is for the skill status/enabled columns, which patchGatewayConfig does NOT touch.

So the correct order is:
1. `patchGatewayConfig()` -- patches gateway config AND persists config to instances table
2. `skills.status` verification -- confirm gateway applied the change
3. `db('instance_skills').update` -- update the skill-specific enabled/status columns

This matches the existing code ordering (RPC then DB update), just replacing the RPC call.
