**mapPluginRow helper**: Maps snake_case DB columns to camelCase InstancePlugin interface, same as mapSkillRow but for `instance_plugins` table. Parse JSON columns (source, config) via `adapter.parseJson()`.

**Exported functions:**

1. `getPluginsForInstance(instanceId)` — SELECT * from instance_plugins WHERE instance_id, ordered by installed_at desc. No lock.

2. `getPluginById(instanceId, pluginId)` — Single row lookup. No lock.

3. `installPlugin(instanceId, pluginId, source, userId)` — Operation 1 from PRD section 5.3:
   - acquireLock(instanceId, 'install', pluginId, 'plugin')
   - INSERT pending row into instance_plugins (id=randomUUID, pending_owner=config.serverSessionId)
   - checkCancelRequested before long operation
   - RPC call: Use `GatewayRPCClient.call('plugins.install', { pluginId, source }, 300_000)` (5-min timeout per INFRA-07 for npm install). This stages the artifact on disk — does NOT touch gateway config.
   - Parse RPC response for version, requiredCredentials
   - If plugin has NO requiredCredentials:
     - Call `activatePlugin(instanceId, pluginId, userId)` within the same lock hold — pass the existing fencingToken to skip re-acquiring. Use an internal `_activatePluginWithLock` helper.
     - This implements PLUG-03 (no-cred skip to activate)
   - If plugin HAS requiredCredentials:
     - UPDATE status -> 'installed', clear pending_owner
     - Return { plugin, requiredCredentials }
   - releaseLock in finally block (always)

4. `activatePlugin(instanceId, pluginId, userId)` — Operation 3 from PRD section 5.3:
   - acquireLock(instanceId, 'activate', pluginId, 'plugin')
   - Call internal `_activatePluginWithLock(instanceId, pluginId, userId, fencingToken, operationId)`
   - releaseLock in finally block

   **_activatePluginWithLock internal helper** (also called from installPlugin for no-cred path):
   - Read plugin row from DB — verify status is 'installed' or 'pending'
   - PLUG-06: Verify artifact exists. If the plugin has a lockedVersion and the artifact is missing (e.g., after rebuild), reinstall from lockedVersion via `GatewayRPCClient.call('plugins.install', { pluginId, version: lockedVersion }, 300_000)`. If reinstall fails, mark failed + release lock + return error.
   - checkCancelRequested before restart
   - Send `config.patch` RPC to add plugin to `plugins.entries.<pluginId>` with `{ enabled: true }` and to `plugins.load.paths` if needed (30s timeout)
   - Call `restartInstance(instanceId, userId)` from instance-manager.ts (this runs seedConfig which picks up the new plugin)
   - Wait for health: After restart, try `GatewayRPCClient.call('platform.ping', {}, 120_000)` (2-min timeout per INFRA-07 for restart)
   - ON SUCCESS: UPDATE status -> 'active', clear pending_owner, set enabled=1
   - ON FAILURE (PLUG-07 rollback):
     - Send `config.patch` to REMOVE this plugin from `plugins.entries.<pluginId>` 
     - Call `restartInstance(instanceId, userId)` again to restore gateway without the broken plugin
     - UPDATE status -> 'failed', set error_message with health check output, set failed_at
     - releaseLock with result='rolled-back'

   IMPORTANT: The activate flow calls restartInstance which triggers seedConfig. The seedConfig in adapter.ts (modified in Task 2) will read instance_plugins for active/degraded plugins. During activation, the plugin's status is still 'installed'/'pending', so seedConfig alone won't pick it up. The config.patch RPC injects it into the running config BEFORE restart. After restart, the new seedConfig should include it because we update the DB status to 'active' AFTER successful health check. But the restart happens before the status update. Solution: Use config.patch to add the plugin entry BEFORE restarting. The gateway loads plugins from the config file which includes both seedConfig defaults and any patches applied. Actually, the simpler approach: update the DB status to 'active' BEFORE calling restartInstance, so that seedConfig picks it up. If health check fails afterward, rollback by setting status back to 'failed' and restarting again. This is safer because seedConfig reads from DB.

   Revised activate flow:
   - UPDATE status -> 'active' in DB
   - Call restartInstance (seedConfig will now include this plugin)
   - Health check via platform.ping
   - ON SUCCESS: done, status already 'active'
   - ON FAILURE: UPDATE status -> 'failed', set error_message; call restartInstance again (seedConfig now excludes the failed plugin); releaseLock with 'rolled-back'

5. `enablePlugin(instanceId, pluginId, userId)` — acquireLock, UPDATE enabled=1 + status='active', call restartInstance (seedConfig picks up the change), releaseLock in finally. 30s RPC timeout for any config.patch if needed, 120s for restart. checkCancelRequested before restart.

6. `disablePlugin(instanceId, pluginId, userId)` — acquireLock, UPDATE enabled=0 + status='disabled', call restartInstance (seedConfig now excludes this plugin), releaseLock in finally. checkCancelRequested before restart.

7. `uninstallPlugin(instanceId, pluginId, userId)` — acquireLock, checkCancelRequested, DELETE from instance_plugins, call restartInstance (seedConfig no longer includes it), releaseLock in finally.

8. `updatePluginStatus(instanceId, pluginId, status, errorMessage?)` — Simple UPDATE helper for reconciliation. No lock needed.

- Server .ts imports MUST use .js extensions (NodeNext ESM resolution)

// Managed plugins from extension management (Phase 2)
// Include active and degraded plugins in seedConfig — degraded may still work from cache
const managedPlugins = await db('instance_plugins')
  .where({ instance_id: instance.id })
  .whereIn('status', ['active', 'degraded'])
  .select('plugin_id', 'config', 'enabled') as Array<Record<string, unknown>>;

for (const mp of managedPlugins) {
  const pluginId = mp.plugin_id as string;
  const pluginConfig = adapter.parseJson<Record<string, unknown>>(mp.config);
  // Only include if enabled (active plugins should always be enabled, but check)
  if (mp.enabled) {
    pluginEntries[pluginId] = { enabled: true, ...pluginConfig };
  }
}
