Security profile config mapping context:
The `seedConfig()` adapter function maps security profiles to full config files. For a security profile change, the delta includes `hooks`, `cron`, and potentially model/approval settings. The simplest approach per research: call `seedConfig()` with the old and new profile to compute the delta, OR have `updateSecurityProfile` pass the security-relevant config keys directly to `patchGatewayConfig`.

Channel credential-to-config resolution context:
Channel configure stores credentials in DB, then needs to push them to the gateway. The credential-to-config mapping is in `seedConfig()`. Per research recommendation: after storing credentials, call the adapter's `seedConfig()` to get the full channel config section, extract the relevant channel delta, and send as merge-patch via `patchGatewayConfig`.

  <name>Task 1: Convert updateSecurityProfile to use patchGatewayConfig</name>
  <files>apps/server/src/services/instance-manager.ts</files>
  <action>
Refactor `updateSecurityProfile()` (lines ~373-428) to delegate to `patchGatewayConfig` for running instances instead of calling `reseedConfigFiles` and `adapter.translateRPC` directly.

**New flow:**

1. Keep the initial `getInstance` check and `db('instances').update({ security_profile, updated_at })` at the top -- the security_profile column is a dedicated column, not part of the config JSON, so this DB write is correct regardless of running state.

2. For running instances (`status === 'running'` and `controlEndpoint` exists):
   - Build a security profile config delta. The simplest correct approach: use the adapter's `seedConfig` to generate the full config for both old and new profiles, then diff them. However, this is complex and `seedConfig` requires many inputs.
   - **Simpler approach (recommended per research):** Since security profile changes affect specific config keys (hooks, cron, etc.), and these are expressed as config changes, build the delta by calling `seedConfig()` once with the new profile and extracting the security-relevant sections (hooks, cron, models, approval settings). Then send just those sections as a merge-patch.
   - **Simplest acceptable approach:** Call `patchGatewayConfig(id, userId, {}, \`Security profile -> ${profile}\`)`. This sends an empty merge-patch which still triggers SIGUSR1 restart (every config.patch does). The gateway will restart and pick up the security_profile change on next boot via `reseedConfigFiles` in `startInstanceAsync`. This is acceptable because: (a) the security_profile column is already updated in DB, (b) the gateway will restart from the config.patch, and (c) on next start, `startInstanceAsync` calls `reseedConfigFiles` which reads the updated security_profile.
   - **However**, an empty merge-patch wastes a rate-limit slot for nothing. Better: extract the adapter's seedConfig output for the security-relevant keys and send those.
   - **DECISION: Use the seedConfig approach.** Call `getAgentType(instance.agentType)` to get the adapter. If the adapter has `seedConfig`, call it with the updated instance (with new profile) to get the config files. Parse the generated `openclaw.json` from the config files map. Extract the top-level keys that are security-profile-dependent: `hooks`, `cron`, `models` (if present). Build a delta from these keys and pass to `patchGatewayConfig`.
   - If `seedConfig` is not available or fails, fall back to sending an empty merge-patch `{}` (still triggers restart).

3. **Remove all old code** from the running-instance block: delete `reseedConfigFiles(id)` call, delete `adapter.translateRPC` calls for config.get and config.patch, delete `engine.readFile` block, delete `getRuntimeEngine` usage.

4. Remove imports that are no longer needed by this function (check if `getRuntimeEngine` is used elsewhere in the file before removing import).

5. Keep the final `return (await getInstance(id, userId))!;` or return the updated instance.

**Error handling:** Wrap the `patchGatewayConfig` call in try/catch. If it fails, log the error but still return the updated instance (security_profile column is already persisted). This matches the current behavior where config.patch failure is non-blocking for the security profile update itself. Add a console.error with `[security-profile]` prefix.

<task type="auto">
  <name>Task 2: Replace reseedAndPatch in channels.ts with patchGatewayConfig</name>
  <files>apps/server/src/routes/channels.ts</files>
  <action>
Replace the `reseedAndPatch()` function and all its call sites in channels.ts with calls to `patchGatewayConfig`. The goal is to send actual channel config deltas through the gateway-first flow instead of writing files to the container volume.

**Changes:**

1. **Delete the `reseedAndPatch` function** (lines 15-45 entirely).

2. **Remove `reseedConfigFiles` from the import** on line 3 (keep `getInstance` and `patchGatewayConfig`).

3. **Add imports** for `getDecryptedCredentials` from `../services/credential-store.js` and `getAgentType` from `../agent-types/registry.js` (getAgentType is already imported on line 3).

4. **Create a helper function** `pushChannelConfigToGateway(instance: Instance, channel: string, userId: string)` that:
   a. Gets decrypted credentials for the instance: `const creds = await getDecryptedCredentials(instance.id);`
   b. Gets the adapter: `const { adapter } = getAgentType(instance.agentType);`
   c. If `adapter?.seedConfig` exists, calls it to generate the full config: `const configFiles = await adapter.seedConfig({ instance, userConfig: instance.config || {}, credentials: creds });`
   d. Extracts the `openclaw.json` content from the configFiles Map: `const openclawJson = configFiles.get('openclaw.json');`
   e. If found, parses it and extracts the `channels` section: `const fullConfig = JSON.parse(openclawJson); const channelDelta: Record<string, unknown> = {};`
   f. Copies the specific channel's config: `if (fullConfig.channels?.[channel]) { channelDelta.channels = { [channel]: fullConfig.channels[channel] }; }`
   g. Also copies the corresponding plugin entry if present: `if (fullConfig.plugins?.entries?.[channel]) { channelDelta.plugins = { entries: { [channel]: fullConfig.plugins.entries[channel] } }; }`
   h. Calls `await patchGatewayConfig(instance.id, userId, channelDelta, \`Platform: configure ${channel}\`);`
   i. If seedConfig is not available or openclawJson not found, fall back to sending empty delta `{}` via patchGatewayConfig (triggers restart, config picked up on next boot).
   j. Wrap entire body in try/catch, log errors with `[channel-config]` prefix (matching existing pattern). Do NOT throw -- channel credential DB changes should succeed even if gateway push fails.

5. **Replace all `reseedAndPatch` call sites:**

   - **Generic channel configure** (~line 474): Replace `await reseedAndPatch(instance, channel);` with `await pushChannelConfigToGateway(instance, channel, req.auth!.userId);`
   
   - **Generic channel disconnect** (~line 512): Replace `await reseedAndPatch(instance, channel);` with `await pushChannelConfigToGateway(instance, channel, req.auth!.userId);` (credentials already deleted, seedConfig will generate config without them)
   
   - **Telegram configure** (~line 549): Replace `await reseedAndPatch(instance, 'telegram');` with `await pushChannelConfigToGateway(instance, 'telegram', req.auth!.userId);`
   
   - **Telegram disconnect** (~line 576): Replace `await reseedAndPatch(instance, 'telegram');` with `await pushChannelConfigToGateway(instance, 'telegram', req.auth!.userId);`

6. All call sites already check `instance.status === 'running' && instance.controlEndpoint` before calling -- keep those guards. `patchGatewayConfig` also checks internally, but the guard prevents unnecessary seedConfig work.

**Important nuance:** For disconnect operations, the credentials have already been deleted from DB BEFORE this function runs. So `seedConfig` will generate config WITHOUT those credentials, and the merge-patch will effectively remove the channel credentials from the gateway config. This is correct behavior.
