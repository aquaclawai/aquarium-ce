Create a new `apps/server/src/routes/oauth-proxy.ts` Express router that proxies the OAuth browser flow for extension-scoped credentials. This route handles the scenario where a plugin requires OAuth authentication (e.g., a plugin declaring it needs GitHub OAuth or Slack OAuth), which is distinct from the existing `oauth.ts` routes that handle instance-level AI provider auth.

**New file: `apps/server/src/routes/oauth-proxy.ts`**

Import: `Router` from `express`, `requireAuth` from `../middleware/auth.js`, `getInstance` from `../services/instance-manager.js`, `addCredential` from `../services/credential-store.js`, `GatewayRPCClient` from `../agent-types/openclaw/gateway-rpc.js`, `randomBytes` from `node:crypto`, `config` from `../config.js`, `db` from `../db/index.js`, types from `@aquarium/shared`.

Create an in-memory state store (same pattern as `oauth.ts` pkceStore) for tracking OAuth state tokens with 10-minute TTL:
```typescript
interface OAuthProxySession {
  instanceId: string;
  extensionId: string;
  extensionKind: ExtensionKind;
  provider: string;
  createdAt: number;
}
const oauthSessions = new Map<string, OAuthProxySession>();
const SESSION_TTL_MS = 10 * 60 * 1000;
```

**Route 1: POST `/:id/oauth-proxy/initiate`**
- Validates instanceId, extensionId, extensionKind, provider from request body
- Verifies instance exists, is running, has controlEndpoint
- Generates a unique state token: `randomBytes(16).toString('hex')`
- Stores session in `oauthSessions` with instance/extension metadata
- Calls gateway RPC method `auth.getOAuthUrl` with params `{ provider, callbackUrl: "${config.corsOrigin}/api/instances/${instanceId}/oauth-proxy/callback?state=${state}" }`. If this RPC method is not supported by the gateway (older versions), fall back to constructing a generic OAuth authorize URL from the provider name and returning it. The RPC timeout is 15 seconds.
- Returns `{ ok: true, data: { authUrl, state } }` so the frontend can redirect/open popup
- On RPC failure: return 502 with descriptive error

**Route 2: GET `/:id/oauth-proxy/callback`**
- Extracts `state` and `code` from query params
- Looks up session from `oauthSessions` by state, validates not expired
- Deletes session from map (one-time use)
- Verifies instance is still running with controlEndpoint
- Relays the authorization code to the gateway via RPC: `auth.exchangeToken` with params `{ provider: session.provider, code, callbackUrl }`. Timeout 30 seconds.
- On success, the gateway handles token storage in `auth-profiles.json` internally
- **CRITICAL: Write an `oauth_token` credential row to `instance_credentials`** so that template export can detect OAuth-backed extensions. After successful token exchange RPC, call:
  ```typescript
  await addCredential(
    session.instanceId,
    session.provider,
    'oauth_token',          // credential_type -- must be 'oauth_token', NOT 'oauth'
    'GATEWAY_MANAGED',      // value -- actual token lives in gateway's auth-profiles.json
    {
      extensionId: session.extensionId,
      extensionKind: session.extensionKind,
    },
  );
  ```
  This row is the sentinel that `template-store.ts` export queries to detect OAuth extensions (Task 2 builds a Set from `credential_type === 'oauth_token'` rows). Without this row, `requiresReAuth` would never be set on export.
- Updates the extension's DB status from `installed` to `active` if it was in `installed` state:
  - If session.extensionKind is `plugin`: `UPDATE instance_plugins SET status='active', updated_at=now() WHERE instance_id=? AND plugin_id=? AND status='installed'`
  - If session.extensionKind is `skill`: same pattern for instance_skills
- Returns an HTML page that posts a message to the opener window and closes itself: `<script>window.opener?.postMessage({type:'oauth-callback',status:'success',extensionId:'...'}, '*'); window.close();</script>`. This allows the CredentialConfigPanel popup flow to detect completion.
- On error (expired state, RPC failure, missing code): return HTML page with error message and auto-close script

**Route 3: GET `/:id/oauth-proxy/status/:extensionId`**
- Quick endpoint for the frontend to poll whether an extension has been OAuth-connected
- Reads the extension's status from instance_plugins or instance_skills
- Returns `{ ok: true, data: { connected: status === 'active' } }`

Apply `requireAuth` middleware to the router. Export default router.

The `adapter.ts` seedConfig later reads `credMeta.source === 'vault'` and `credMeta.vaultPath` to resolve vault references -- so they MUST be persisted.

**Update template-store.ts `exportFromInstance` (around lines 603-665):**

After reading `instance_plugins` and `instance_skills`, and before pushing to the `extensions` array, check whether each extension has OAuth-type credentials. The detection strategy:

1. After fetching `pluginRows` and `skillRows`, also fetch the instance's credentials:
   ```typescript
   const instanceCreds = await db('instance_credentials')
     .where({ instance_id: instanceId })
     .select('provider', 'credential_type', 'metadata');
   ```

2. Build a Set of extension IDs that have OAuth credentials. Parse each credential's `metadata` JSON column to check for `extensionId` and `extensionKind` fields. Also check `credential_type === 'oauth_token'`:
   ```typescript
   const oauthExtensionIds = new Set<string>();
   for (const cred of instanceCreds) {
     if (cred.credential_type === 'oauth_token') {
       const meta = typeof cred.metadata === 'string' ? JSON.parse(cred.metadata) : (cred.metadata ?? {});
       if (meta.extensionId) {
         oauthExtensionIds.add(meta.extensionId);
       }
       // Also add by provider name as fallback (provider may match extensionId for extension-scoped creds)
       oauthExtensionIds.add(cred.provider);
     }
   }
   ```

   NOTE: This depends on Task 1's `addCredential()` call in the OAuth callback writing a row with `credential_type = 'oauth_token'` and `metadata.extensionId`. Without that sentinel row, this Set would always be empty and no extensions would get `requiresReAuth: true`.

3. When building each `TemplateExtensionDeclaration`, set `requiresReAuth: true` if the extension's ID is in `oauthExtensionIds`:
   ```typescript
   // In the plugin loop:
   extensions.push({
     ...existingFields,
     requiresReAuth: oauthExtensionIds.has(row.plugin_id as string),
   });
   // In the skill loop:
   extensions.push({
     ...existingFields,
     requiresReAuth: oauthExtensionIds.has(row.skill_id as string),
   });
   ```

4. For extensions with `requiresReAuth: true`, also force `needsCredentials: true` and clear any OAuth-specific config values from the `config` field to prevent token leakage. If `requiresReAuth` is false or undefined, omit the field (don't serialize `requiresReAuth: false`).
