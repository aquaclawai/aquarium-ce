Extend the `seedConfig` function in adapter.ts to support `exec` SecretRef for extension credentials when the instance has a vault configured.

**How exec SecretRef works:**
When an extension credential's metadata contains `{ source: 'vault', vaultPath: 'secret/data/openai/apiKey' }`, instead of generating an `env`-backed SecretRef (`{ source: 'env', id: 'AQUARIUM_CRED_xxx' }`), generate an `exec`-backed SecretRef that tells the gateway to run a CLI command at startup:

For 1Password: `{ source: 'exec', command: ['op', 'read', 'op://<vaultPath>'] }`
For HashiCorp: `{ source: 'exec', command: ['vault', 'kv', 'get', '-field=value', '-mount=<mountPath>', '<vaultPath>'] }`

**Implementation steps:**

1. In `seedConfig`, after reading instance config, extract `vaultConfig`:
   ```typescript
   const vaultConfig = (userConfig as Record<string, unknown>).vaultConfig as {
     type: 'onepassword' | 'hashicorp';
     address?: string;
     namespace?: string;
     mountPath?: string;
   } | undefined;
   ```

2. In the managed extension credential injection section (where `AQUARIUM_CRED_<hash>` env-backed SecretRefs are built), check each extension credential's metadata for a `source: 'vault'` indicator:
   ```typescript
   // Inside the extension credential resolution loop
   const credMeta = /* parsed credential metadata */;
   if (credMeta?.source === 'vault' && credMeta?.vaultPath && vaultConfig) {
     // Use exec SecretRef instead of env SecretRef
     let execCommand: string[];
     if (vaultConfig.type === 'onepassword') {
       execCommand = ['op', 'read', `op://${credMeta.vaultPath}`];
     } else {
       const mount = vaultConfig.mountPath || 'secret';
       execCommand = ['vault', 'kv', 'get', '-field=value', `-mount=${mount}`, credMeta.vaultPath as string];
     }
     secretRef = { source: 'exec', command: execCommand };
   }
   ```
   For vault-backed credentials, do NOT add to the `env` map (no `AQUARIUM_CRED_xxx` env var needed -- the gateway resolves via CLI at runtime).

3. Update the `secrets.providers` config section. When vault is configured, add the vault provider alongside the default env provider:
   ```typescript
   if (vaultConfig) {
     cfg.secrets = {
       providers: {
         default: { source: 'env' },
         vault: { source: 'exec' },
       },
     };
     // For HashiCorp, also inject VAULT_ADDR and optionally VAULT_NAMESPACE into env
     if (vaultConfig.type === 'hashicorp' && vaultConfig.address) {
       env['VAULT_ADDR'] = vaultConfig.address;
       if (vaultConfig.namespace) {
         env['VAULT_NAMESPACE'] = vaultConfig.namespace;
       }
     }
   }
   ```

4. In `resolveEnv`, skip injecting `AQUARIUM_CRED_<hash>` env vars for credentials that have `source: 'vault'` in their metadata. These are resolved by the gateway via exec, not env vars.

The implementation must handle the case where some credentials are vault-backed and others are env-backed for the same instance (mixed mode). Each credential is resolved independently based on its metadata.

Guard this behind `supportsSecretRef(instance.imageTag)` -- older gateway images that don't support SecretRef at all cannot use vault integration.
