  **Rewrite `cancelPendingTasksForIssueAgent`** (preserve existing queued|dispatched scope; add row collection + optional broadcast):

  ```typescript
  export async function cancelPendingTasksForIssueAgent(
    args: CancelPendingArgs & { emitBroadcasts?: boolean },
  ): Promise<CancelResult> {
    const { workspaceId, issueId, agentId, trx, emitBroadcasts } = args;

    // Collect-then-update pattern to capture previous statuses + ids for broadcast.
    const doWork = async (t: Knex.Transaction | typeof db): Promise<CancelResult> => {
      const rows = await t('agent_task_queue')
        .where({ workspace_id: workspaceId, issue_id: issueId, agent_id: agentId })
        .whereIn('status', ['queued', 'dispatched'])
        .select('id', 'status', 'issue_id', 'workspace_id');
      if (rows.length === 0) return { count: 0, cancelledTaskIds: [], cancelledRows: [] };

      const ids = rows.map((r) => r.id as string);
      await t('agent_task_queue')
        .whereIn('id', ids)
        .whereIn('status', ['queued', 'dispatched']) // ST6 race guard
        .update({ status: 'cancelled', cancelled_at: db.fn.now(), updated_at: db.fn.now() });

      return {
        count: ids.length,
        cancelledTaskIds: ids,
        cancelledRows: rows.map((r) => ({
          taskId: r.id as string,
          issueId: r.issue_id as string,
          workspaceId: r.workspace_id as string,
          previousStatus: r.status as TaskStatus,
        })),
      };
    };

    // If caller provided trx, run inside it; caller handles broadcasts after commit.
    let result: CancelResult;
    if (trx) {
      result = await doWork(trx);
    } else {
      result = await db.transaction(async (t) => {
        await t.raw('BEGIN IMMEDIATE');
        return doWork(t);
      });
      // Only broadcast here if caller asked AND no trx was supplied (internal commit done).
      if (emitBroadcasts) {
        for (const r of result.cancelledRows) {
          broadcast(r.workspaceId, {
            type: 'task:cancelled',
            taskId: r.taskId,
            issueId: r.issueId,
            payload: { taskId: r.taskId, issueId: r.issueId },
          });
        }
      }
    }
    return result;
  }
  ```

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-18-19 | Tampering (ghost broadcast) | caller rolls back their transaction but a `task:cancelled` event was already emitted | mitigate | Helpers with `trx` argument do NOT broadcast — they return `cancelledRows` and let the caller broadcast AFTER its own commit in the route handler (matches Phase-17 pattern for `issue:*` events). Without-trx path broadcasts only after the internal `db.transaction(...)` commit. |
| T-18-20 | Information disclosure (WS leak across workspace) | task:cancelled for workspace X delivered to client subscribed to workspace Y | mitigate | `broadcast(row.workspaceId, ...)` uses the row's own workspace_id — same isolation boundary as Phase 17's `issue:*`. CE is single-workspace (`'AQ'`) so in practice this is a future-proofing for EE. |
| T-18-21 | Tampering (reassignment race clobbers cancel) | user reassigns issue while another thread cancels it; which transition wins? | mitigate | Both branches run inside the same `updateIssue` trx — serialised under pool=1. The mutually-exclusive decision ladder in `applyIssueSideEffects` runs cancelled-transition FIRST and returns; reassignment only fires if next.status !== 'cancelled'. Phase 17 already enforces this; 18-04 inherits. |
| T-18-22 | DoS (broadcast storm from large cancelAll) | cancelling an issue with 100+ running tasks fan-outs 100 WS events in a tight loop | accept | CE single-user load makes this realistic cap ~20 tasks per issue; fan-out is synchronous over the `clients` Set (in-process) — cost is microseconds per event. Defer WS backpressure to v1.5 if real-world usage shows issues. |
| T-18-23 | Tampering (discard-after-cancel stays silent) | daemon posts completeTask(id, result) 1ms after cancel; completeTask returns `{ discarded: true }` — but does NOT broadcast task:completed (correct!). Does this leave the daemon confused? | accept | PM5 says daemon must treat `{ discarded: true }` as a non-error discard and stop reporting. Phase 19 HTTP route maps discarded -> 200 OK so daemon's error handler does not fire. No broadcast is correct: the UI is already cancelled and does not need a phantom completion event. |
| T-18-24 | Spoofing (cross-workspace cancel) | user in workspace A sends PATCH /api/issues/:id on an issue owned by workspace B | mitigate | Phase 17 route handler already filters by `DEFAULT_WORKSPACE_ID='AQ'` in CE; the updateIssue service loads the issue with `.where({ id, workspace_id: workspaceId })`. Phase 18-04 does not touch this boundary. |
