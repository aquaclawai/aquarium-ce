Add ping/pong heartbeat to the PersistentGatewayClient class in gateway-event-relay.ts:

1. Add two private fields to the class:
   - `private pingTimer: ReturnType<typeof setInterval> | null = null;`
   - `private lastPongAt: number = 0;`

2. Add `startPingLoop()` private method:
   - Call `stopPingLoop()` first (idempotent cleanup).
   - Set `this.pingTimer = setInterval(...)` with 30_000ms interval.
   - In the interval callback:
     - If `!this.ws || !this.connected` return early.
     - Compute `elapsed = Date.now() - this.lastPongAt`.
     - If `elapsed > 60_000`: log warning `[gateway-relay] No pong from ${this.instanceId} for ${elapsed}ms, forcing reconnect`, then call `this.ws.terminate()` (NOT `this.ws.close()` -- terminate destroys immediately without waiting for close handshake, correct for frozen peer). Return after terminate.
     - Otherwise call `this.ws.ping()`.

3. Add `stopPingLoop()` private method:
   - If `this.pingTimer` is set, `clearInterval(this.pingTimer)` and set to null.

4. Wire into connection lifecycle -- in the `connect()` method:
   - At the very start of `connect()` (after `if (this.closed) return;`), call `this.stopPingLoop()` to clean up any stale timer from a previous connection.
   - In the message handler, right after `this.connected = true;` and `this.retryCount = 0;` (the hello-ok success path, before the syncGatewayState call):
     - Set `this.lastPongAt = Date.now();`
     - Register: `ws.on('pong', () => { this.lastPongAt = Date.now(); });`
     - Call `this.startPingLoop();`

5. Wire cleanup into `ws.on('close')` handler:
   - Call `this.stopPingLoop()` at the start of the close handler (before rejecting pending requests).

6. Wire cleanup into `close()` method:
   - Call `this.stopPingLoop()` near the top of `close()` (after setting `this.closed = true`, alongside the existing `retryTimeout` and `restartTimer` cleanup).

IMPORTANT: The `pong` listener must be registered on the specific `ws` instance created in `connect()` (the local `ws` variable that is captured in the closure), not on `this.ws` (which may be reassigned). This matches how the existing `message`, `close`, and `error` handlers are registered.
