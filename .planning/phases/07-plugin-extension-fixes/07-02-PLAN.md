**FRONT-01 fix -- Response destructuring** (6 locations in ExtensionsTab.tsx):
The catalog endpoints return `{ catalog: T[], hasMore: boolean }` but the old code expected a flat `T[]` array. All 6 locations are fixed:
1. Initial skill catalog fetch: `api.get<{ catalog: SkillCatalogEntry[]; hasMore: boolean }>` then `setCatalog(catalogData.catalog)` and `setClawHubSkillHasMore(catalogData.hasMore)`
2. Initial plugin catalog fetch: same pattern with `PluginCatalogEntry`
3. Load more plugins: `setPluginCatalog(prev => [...prev, ...moreData.catalog])` and `setClawHubPluginHasMore(moreData.hasMore)`
4. Load more skills: `setCatalog(prev => [...prev, ...moreData.catalog])` and `setClawHubSkillHasMore(moreData.hasMore)`

**FRONT-02 fix -- Source object format** (3 locations):
Install handlers now send `source: { type: 'clawhub', spec: id }` (object) instead of `source: 'clawhub'` (string):
1. `handleInstallSkill`: creates `sourceObj` from source string, sends as object
2. `handleInstallPlugin`: creates `sourceObj` from source string, sends as object
3. Failed skill retry button: sends `source: { type: 'bundled' }` instead of `source: 'bundled'`

3. Type annotations match the actual response shape
4. No raw `fetch()` calls (uses `api.get`/`api.post` per project convention)
5. No hardcoded colors or missing i18n strings

If any issue is found during review, fix it. If all changes look correct, they are ready.
