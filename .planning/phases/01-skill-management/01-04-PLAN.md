<task type="auto">
  <name>Task 1: Create extension-lifecycle service with reconciliation and orphan recovery</name>
  <files>apps/server/src/services/extension-lifecycle.ts</files>
  <action>
Create `apps/server/src/services/extension-lifecycle.ts` implementing two main functions.

Import: `db` from `../db/index.js`, `config` from `../config.js`, `getAdapter` from `../db/adapter.js`, `GatewayRPCClient` from `../agent-types/openclaw/gateway-rpc.js`, `cleanupOrphanedOperations` from `./extension-lock.js`, `getSkillsForInstance`, `updateSkillStatus` from `./skill-store.js`, types from `@aquarium/shared`.

**Exported functions:**

1. `recoverOrphanedOperations(): Promise<void>`
   - Call `cleanupOrphanedOperations()` — marks stale-session ops as crashed
   - SELECT all `instance_skills` WHERE `pending_owner IS NOT NULL` AND `pending_owner != config.serverSessionId` AND `status = 'pending'`
   - These are skills whose install was in-progress when the previous server crashed. Leave them as 'pending' — reconciliation (Phase 2) or pending replay (Phase 3) will resolve them.
   - Log count of orphaned operations and orphaned pending extensions found
   - This function should be called once on server startup (before any instance starts)

2. `reconcileExtensions(instanceId: string, controlEndpoint: string, authToken: string): Promise<{ promoted: string[]; demoted: string[]; unchanged: string[] }>`
   - This implements PRD section 5.4, Phase 2 reconciliation
   - Call `rpc.call('skills.list')` to get current gateway skill state
   - Call `getSkillsForInstance(instanceId)` to get DB state
   - Build maps for comparison:
     - `gatewaySkills`: Map<skillId, gatewaySkillInfo> from RPC response
     - `dbSkills`: Map<skillId, InstanceSkill> from DB
   
   - For each DB skill:
     a. **active/degraded in DB, present in gateway**: No change (confirmed healthy). If was 'degraded', promote to 'active'. Add to `promoted` or `unchanged`.
     b. **active in DB, absent from gateway**: Mark 'failed' with error "Extension not found in gateway after restart". Add to `demoted`.
     c. **degraded in DB, absent from gateway**: Mark 'failed' with error "Extension not recovered after restart". Add to `demoted`.
     d. **pending in DB, present in gateway** (crash recovery case): Promote to 'active' — the install completed before the crash. Add to `promoted`. Clear pending_owner.
     e. **pending in DB, absent from gateway**: Leave as 'pending' for Phase 3 replay. Add to `unchanged`.
     f. **installed/disabled/failed in DB**: No reconciliation needed. Add to `unchanged`.
   
   - For each gateway skill NOT in DB (built-in skills):
     - Do NOT insert into DB (per PRD section 5.7: "Gateway-only extensions are not inserted into DB")
     - These are returned via the GET /skills endpoint as `gatewayBuiltins`
   
   - Log reconciliation summary: promoted, demoted, unchanged counts
   - Return `{ promoted, demoted, unchanged }` (arrays of skillId strings)

3. `getPendingExtensionsForReplay(instanceId: string): Promise<InstanceSkill[]>`
   - SELECT FROM instance_skills WHERE instance_id = instanceId AND status = 'pending'
   - Return mapped rows — these are candidates for Phase 3 replay
   - Phase 3 implementation (actually replaying installs) will be handled by the adapter integration
