**--- (A) Server-side trust enforcement on install endpoints (TRUST-03) ---**

**plugins.ts -- modify POST /:id/plugins/install:**

After validating the source object and before calling installPlugin(), add a trust policy check:

1. If source.type is NOT 'bundled' (i.e., 'clawhub' or 'npm'):
   - Fetch trust signals: call getClawHubExtensionInfo(controlEndpoint, authToken, pluginId, 'plugin') to get ClawHub metadata (may be null if ClawHub unreachable)
   - Extract signals: `const signals = clawHubInfo?.trustSignals ?? null;`
   - Call `const evaluation = await evaluateTrustPolicy(instance.id, pluginId, 'plugin', pluginSource, signals);`
   - If evaluation.decision === 'block': return 403 `{ ok: false, error: evaluation.blockReason ?? 'Extension blocked by trust policy' }`
2. If source.type === 'bundled': skip trust check (bundled always allowed)

This ensures even if the UI is bypassed (e.g., API call), the server blocks untrusted installs.

**skills.ts -- modify POST /:id/skills/install:**

Same pattern:
1. If source.type is NOT 'bundled':
   - Fetch ClawHub info for the skill
   - Call evaluateTrustPolicy(instance.id, skillId, 'skill', skillSource, signals)
   - If decision === 'block': return 403 with blockReason
2. If source.type === 'bundled': skip trust check

**--- (B) Catalog endpoints with ClawHub merging and per-entry trust evaluation ---**

**plugins.ts -- extend GET /:id/plugins/catalog:**

1. Read query params: `search`, `category`, `page` (default '0'), `limit` (default '20')
2. Existing code fetches bundled catalog from `plugins.list` RPC. Keep that as-is.
3. After bundled results, if instance is running and controlEndpoint exists, also call `searchClawHub(controlEndpoint, authToken, { query: search, category, kind: 'plugin', offset: page * limit, limit })` from marketplace-client.ts
4. For each bundled catalog entry, set `trustTier: 'bundled' as TrustTier`, `trustDecision: 'allow' as TrustDecision`, `blockReason: null`, `trustSignals: undefined`
5. **For each ClawHub entry, call `evaluateTrustPolicy(instance.id, entry.id, 'plugin', { type: 'clawhub', spec: entry.id }, entry.trustSignals)` and set:**
   - `trustTier: evaluation.tier`
   - `trustDecision: evaluation.decision`
   - `blockReason: evaluation.blockReason`
   - `trustSignals: entry.trustSignals`
   Map each ClawHub entry to PluginCatalogEntry shape with these added fields.
6. Merge: bundled entries first, then ClawHub entries. Deduplicate by id (bundled wins if same id exists in both)
7. Return merged list in existing ApiResponse format, including `hasMore` from ClawHub response
8. If searchClawHub fails, soft-log warning and return only bundled results (same resilience pattern as plugins.list RPC failure)

**skills.ts -- same pattern:**

1. Extend GET /:id/skills/catalog with ClawHub merging
2. For each ClawHub entry, call `evaluateTrustPolicy(instance.id, entry.id, 'skill', { type: 'clawhub', spec: entry.id }, entry.trustSignals)` to compute trustDecision and blockReason per entry
3. Merge bundled + ClawHub, deduplicate, return with hasMore

**--- (C) Upgrade endpoints with dryRun support ---**

**plugins.ts -- add PUT /:id/plugins/:pluginId/upgrade:**

IMPORTANT: Define this route BEFORE the existing PUT /:id/plugins/:pluginId route to prevent Express route capture. The path is `/:id/plugins/:pluginId/upgrade`.

1. Verify instance exists, belongs to user, is running
2. Read existing plugin row via getPluginById -- must exist and be in 'active' or 'installed' or 'disabled' status
3. Read `dryRun` from req.body (boolean, default false)
4. Call getClawHubExtensionInfo(controlEndpoint, authToken, pluginId, 'plugin') to get latest version
5. If no ClawHub info or same version as lockedVersion: return 200 with `{ ok: true, data: { upToDate: true, currentVersion: plugin.lockedVersion } }`
6. **If dryRun === true:** return 200 with `{ ok: true, data: { upToDate: false, currentVersion: plugin.lockedVersion, newVersion: clawHubInfo.version } }` -- do NOT perform install
7. **If dryRun is false or omitted AND newer version available:** acquire lock, call plugins.install RPC with the new version, update locked_version and integrity_hash in DB, release lock
8. If plugin was 'active': trigger activation (restart) for the new version
9. Return 200 with `{ ok: true, data: { upToDate: false, previousVersion, newVersion, plugin } }`

**skills.ts -- add PUT /:id/skills/:skillId/upgrade:**

Same pattern:
1. Similar to plugin upgrade but NO restart needed (skills don't require restart)
2. Support `dryRun` in req.body: if true, return version comparison without performing install
3. If dryRun is false or omitted: acquire lock, call skills.install RPC with new version spec, update locked_version and integrity_hash, release lock
4. Define this route BEFORE the existing PUT /:id/skills/:skillId route
