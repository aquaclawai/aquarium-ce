  <name>Task 2: Version pinning and integrity hash in skill-store and plugin-store install flows</name>
  <files>apps/server/src/services/skill-store.ts, apps/server/src/services/plugin-store.ts</files>
  <action>
Extend both store services to pin version + compute/verify SHA-512 integrity hash on install. Import `createHash` from 'node:crypto'.

**skill-store.ts changes:**

1. After the successful `skills.install` RPC response (around line 164-181), extract version and integrityHash from the RPC result:
   - Add `version` and `integrityHash` to the `isInstallRPCResult` type guard (both optional strings from RPC)
   - The RPC result shape: `{ requiredCredentials?: [...], version?: string, integrityHash?: string }`
   - If rpcResult.version exists, pin it as locked_version
   - If rpcResult.integrityHash exists, store it as integrity_hash
   - Update the DB UPDATE statement (line 175-181) to include `version`, `locked_version`, and `integrity_hash`

2. If the RPC does NOT return integrityHash but returns a version, store locked_version but leave integrity_hash null (the gateway may not support hash computation yet -- defensive).

3. **Integrity verification on skill reinstall (TRUST-06):** When installSkill is called for a skill that already exists in the DB (i.e., reinstall path -- the skill row already has a `locked_version` and `integrity_hash` from a previous install):
   - Before the RPC call, read the existing skill row to get the current `locked_version` and `integrity_hash`
   - After the RPC returns with a new integrityHash, if the existing row has BOTH a non-null `locked_version` and non-null `integrity_hash`, AND the RPC result version matches the existing locked_version (same version reinstall), compare the new integrityHash with the stored integrity_hash
   - If mismatch: throw new Error(`Integrity mismatch -- registry returned different artifact for v${lockedVersion}. Possible supply-chain tampering. Contact the extension publisher.`)
   - This causes the reinstall to fail with the integrity error propagated to the caller
   - If the RPC result version is different from locked_version (explicit upgrade), skip the check -- the hash is expected to differ
   - If no stored integrity_hash (legacy installs or first install), skip verification

**plugin-store.ts changes:**

1. After the successful `plugins.install` RPC response (around line 308-327), the code already stores `version` and `locked_version`. Extend to also store `integrity_hash`:
   - Add `integrityHash` to the `isInstallPluginRPCResult` type guard
   - If result.integrityHash exists, include it in the DB update alongside locked_version

2. In `_activatePluginWithLock` (the activation flow around line 394+), add integrity verification:
   - Before proceeding with activation, read the stored `integrity_hash` from the plugin row
   - If the plugin has a `locked_version` and an `integrity_hash`, and activation involves a reinstall (PLUG-06 artifact missing check), verify the newly installed artifact's hash matches:
     - After reinstall RPC returns, compare rpcResult.integrityHash with stored integrity_hash
     - If mismatch: throw new Error(`Integrity mismatch -- registry returned different artifact for v${lockedVersion}. Possible supply-chain tampering. Contact the extension publisher.`)
     - This causes the activation to fail, marking the plugin as 'failed' with the integrity mismatch error message
   - If no stored integrity_hash (legacy installs), skip verification

IMPORTANT: Do NOT change the function signatures -- only the internal implementation. The integrity mismatch error message must match the CONTEXT.md spec exactly: "Integrity mismatch -- registry returned different artifact for v{version}. Possible supply-chain tampering. Contact the extension publisher."
