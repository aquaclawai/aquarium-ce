<objective>
Extend template import (instantiateTemplate) to re-evaluate trust policy for each extension declaration against current ClawHub metadata, insert lifecycle rows for trusted extensions, omit blocked extensions with warnings, and update setup command generation to skip lifecycle-managed extensions.

Purpose: When a template is imported, the trust landscape may have changed since export. Extensions that were safe at export time may now be compromised, revoked, or unverified. Trust re-evaluation at import time ensures the target instance only gets extensions that pass current security policy.

Output: Modified instantiateTemplate that processes TemplateExtensionDeclaration[], evaluates trust, inserts lifecycle rows, and returns blocked extension warnings.
</objective>

2. In `instantiateTemplate`, after the instance is created (line ~810, after `await db('instances')...update`) and before the credential store loop, add extension processing:

   a. Read extensions from template. The template content (from Plan 04-01) now includes `extensions` in `template_contents`. Parse it:
   ```typescript
   const templateExtensions = content
     ? parseJsonb<TemplateExtensionDeclaration[]>(content.extensions, [])
     : [];
   // Also check legacy: if templateExtensions is empty, fall back to template.plugin_dependencies + template.skills
   ```

   If `templateExtensions` is empty, fall back to existing behavior (generateDependencySetupCommands handles MCP + legacy).

   b. For each extension in `templateExtensions`, perform trust re-evaluation (TMPL-06):
   ```typescript
   const blockedExtensions: Array<{ id: string; kind: ExtensionKind; reason: string }> = [];
   const requiresTrustOverride: Array<{ id: string; kind: ExtensionKind; source: PluginSource | ExtensionSkillSource; reason: string }> = [];
   let extensionsImported = 0;
   ```

   For each extension:
   - If `source.type === 'bundled'`: always passes, skip trust check
   - Otherwise: The instance is not running yet (just created), so we cannot use RPC to fetch ClawHub info. Instead, check the source type:
     - `bundled`: always allow
     - `clawhub` or `npm`: evaluate trust. Since we can't call the gateway RPC (instance not running), use `evaluateTrustPolicy(instance.id, ext.id, ext.kind, ext.source, null)`. Passing `null` for signals means the trust store will check for existing admin overrides only. For `verified` sources that need current metadata, we can't verify at import time (gateway not running). Instead, insert as `pending` and let Phase 3 replay handle the actual install which will perform trust checks.
     - ALTERNATIVE (simpler): Since the gateway is not running, skip the full trust evaluation for non-bundled extensions. Instead, categorize by source type:
       - `bundled`: insert directly as `pending`/`disabled`
       - `clawhub`/`npm` with lockedVersion+integrityHash: insert as `pending` -- Phase 3 replay will perform trust evaluation when the gateway is running and can query ClawHub
       - `clawhub`/`npm` without locked version: still insert as `pending` for Phase 3 handling
     - NOTE: The trust re-evaluation happens effectively at Phase 3 install time, since installSkill/installPlugin already call evaluateTrustPolicy internally. This is the correct approach because the gateway must be running to query ClawHub for trust signals.
     - For the import response: we cannot determine blocked vs allowed at import time without the gateway. Return all non-bundled extensions in `requiresTrustOverride` as informational (the UI can show "trust will be verified on first boot").

   c. Insert lifecycle rows for each extension that passes (TMPL-07):
   - For plugins (`kind === 'plugin'`): INSERT into `instance_plugins` with:
     - `id`: randomUUID()
     - `instance_id`: instance.id
     - `plugin_id`: ext.id
     - `source`: JSON.stringify(ext.source)
     - `locked_version`: ext.lockedVersion
     - `integrity_hash`: ext.integrityHash
     - `enabled`: ext.enabled
     - `config`: JSON.stringify(ext.config ?? {})
     - `status`: ext.enabled ? 'pending' : 'disabled'
     - `installed_at`: db.fn.now()
     - `updated_at`: db.fn.now()

   - For skills (`kind === 'skill'`): INSERT into `instance_skills` with same pattern but `skill_id` instead of `plugin_id`

   - Increment `extensionsImported` for each successful insert

   d. Update `generateDependencySetupCommands` usage: If `templateExtensions.length > 0`, skip plugins and skills in `generateDependencySetupCommands` -- only pass MCP servers. The lifecycle rows handle plugin/skill installation via Phase 3 startup:
   ```typescript
   const autoSetupCommands = templateExtensions.length > 0
     ? generateDependencySetupCommands([], [], parseJsonb<Record<string, McpServerDeclaration>>(template.mcp_servers, {}))
     : generateDependencySetupCommands(
         parseJsonb<SkillDeclaration[]>(template.skills, []),
         parseJsonb<PluginDependency[]>(template.plugin_dependencies, []),
         parseJsonb<Record<string, McpServerDeclaration>>(template.mcp_servers, {}),
       );
   ```

4. NOTE on the `extensions` field in template_contents: Plan 04-01 adds extensions to the ExportTemplateResponse content and template-file-format. But `template_contents` in the DB stores content as individual JSONB columns (workspace_files, mcp_server_configs, etc.), NOT as the TemplateContent interface directly. The extensions need to either:
   - Be stored as a new `extensions` JSONB column in `template_contents` -- but this requires a migration
   - OR be stored in the template manifest (templates table) as a JSONB column
   - OR be parsed from the template's plugin_dependencies/skills fields which already exist

   The simplest approach: When a template is created from export (the createTemplate or updateTemplate flow), store the extensions array in the `plugin_dependencies` field of `template_contents`, overloading it with the new format. The `parseJsonb` will handle both old PluginDependency[] and new TemplateExtensionDeclaration[] arrays. Discriminate by checking for the `kind` field:
   ```typescript
   const rawExtData = parseJsonb<unknown[]>(content.plugin_dependencies, []);
   const templateExtensions: TemplateExtensionDeclaration[] = rawExtData
     .filter((item): item is TemplateExtensionDeclaration =>
       typeof item === 'object' && item !== null && 'kind' in item);
   ```
   If `kind` field is absent, treat as legacy PluginDependency[].

   ALTERNATIVELY, check if the template was imported via .octemplate (which has the new `extensions` field in template.json). In that case, the `parseOctemplate` result from Plan 04-01 already puts extensions into `content.extensions`. When saving to DB via createTemplate, store extensions in a JSON field. The cleanest approach is to use the existing `plugin_dependencies` column in `template_contents` to store the full extensions array when extensions exist, since this column is already JSONB and the new format is a superset.

   Document the approach decision in the code with a comment explaining the format discrimination.
